### NIT (2)

**NIT-1: Redundant `vv_before`/`vv_after` assertion in `ordered_collection_reorder_preserves_identity`**

**Location**: `tests/unit/schema_roundtrip.rs:223` â€” `assert_ne!(vv_before, vv_after, "oplog vv must advance after reorder");`

**Issue**: If `vv_before == vv_after`, then `doc.diff(&f_before, &f_after)` would return an empty `DiffBatch`, and the `saw_move` assertion on line 246 would fail with a clearer error message ("expected at least one Move op"). The `assert_ne!` is therefore somewhat redundant with the diff inspection.

**Mitigation**: The `assert_ne!` provides a clearer precondition error message ("oplog vv must advance after reorder") vs the diff inspection's failure message. Acceptable as-is.

**Concrete fix**: No action â€” leave as-is. The redundancy provides defense-in-depth (precondition check + shape check).

**NIT-2: `tree_node_flat_roundtrip` uses `ExportMode::all_updates()` instead of incremental `ExportMode::updates(&vv)`**

**Location**: `tests/unit/schema_roundtrip.rs:292,321,322` â€” `doc.export(ExportMode::all_updates())`.

**Issue**: `all_updates()` exports ALL ops from the beginning of the doc, not just the new ones since the last sync. This works because Loro is content-addressed and idempotent for already-seen ops (re-importing an already-applied op is a no-op). Production sync would use `ExportMode::updates(&vv)` for incremental sync.

**Mitigation**: The test's goal is to verify field-level concurrent merge convergence, NOT sync efficiency. The `all_updates()` path is simpler and tests the convergence property equally well. Acceptable for a unit test.

**Concrete fix**: No action â€” leave as-is. If a future Phase adds an incremental-sync integration test, that test should use `ExportMode::updates(&vv)`.

### ACCEPTABLE (1)

**ACCEPTABLE-1: L3 documented API deviation from L2 handoff**

**Verdict**: ACCEPTABLE. The deviation is documented in the worklog with citation. No hallucination â€” the alternative API is verified against crate source. The worklog note "future L1/L2 agents writing similar tests should reference `doc.diff()` + `doc.oplog_frontiers()`, not the nonexistent `export_from`" is accurate and useful.
